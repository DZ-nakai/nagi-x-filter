# 設計判断メモ (Design Decisions)

このドキュメントは「なぜこの構成にしたか」を記録するもの。
ポートフォリオとして、課題発見 → 技術選定 → トレードオフ判断の流れを示すことを目的とする。

## 0. 解きたい課題

X の検索（特に「最新 / Latest」タブ）で、自分がブロック / ミュートしたはずの
アカウントの投稿がタグ検索などで表示されてしまう。X 標準の
「ブロックとミュートのアカウントを除外」設定は存在するが、取りこぼしがある。
→ **クライアント側でその穴を埋める。**

## 1. なぜ「ブラウザ拡張」ではなく「ユーザースクリプト」か

| 候補 | PC | iOS | Android | 1コードで全対応 |
|---|---|---|---|---|
| Chrome 拡張専用 | ◎ | △(Orion等) | △(Kiwi等) | × |
| **ユーザースクリプト** | ◎(Tampermonkey) | ◎(Safari+Userscripts) | ◎(Firefox+Violentmonkey) | **◎** |

モバイル対応を要件に含めた結果、**1 本の `.user.js` で PC / iOS / Android を
カバーできるユーザースクリプト**を選択。拡張専用にするとモバイルへ展開できない。

## 2. なぜ公式 API を使わないか

- X API v2 の取得系（blocking/muting）は有料階層が前提で、月額コストが大きい。
- 収益化しない個人ツールに API 課金は見合わない。
- ユーザースクリプトは**ログイン済みセッションの中で動く**ため、
  すでに画面に出ている情報を読むだけで完結できる。

→ **API 非依存・端末内完結**を設計の前提に置く。

## 3. ブロック / ミュート一覧の取得方法

公式 API を使わずに「誰をブロック / ミュートしているか」を知る手段:

- **採用: 設定ページからの収集** — `/settings/blocked/all`・`/settings/muted/all`
  を開いたときに、レンダリング済み DOM から handle を収集して端末内に保存する。
  「自分が閲覧権限を持つページを読むだけ」で、公式 API は一切叩かない。
- 不採用: 手動リスト管理のみ — 確実だが運用負荷が高い（補助手段として残す）。

## 4. 敵対的 DOM への耐性

X はクラス名をランダム化するため、見た目のクラスに依存すると壊れる。

- 構造の手がかりには **`data-testid`** を使う（`article[data-testid="tweet"]` 等）。
- セレクタは **1 箇所（`SELECTORS`）に集約**し、X 側変更時の修正範囲を最小化する。
- handle 抽出は、表示テキストではなく **プロフィールリンクの `href="/handle"`** から
  正規表現で取り出す（最も安定）。
- 仮想スクロール（DOM の付け外し）に追従するため **`MutationObserver`** で監視する。

## 5. プライバシー / データの扱い

- 収集した handle は **`GM_setValue` で端末内のみ**に保存。外部送信は一切なし。
- これは「ユーザーが自分のフィードを制御する」ための補助であり、
  X のデータを収集・再配布するものではない。

## 6. 既知の限界とトレードオフ

- X の DOM 変更により、セレクタの追従メンテが定期的に必要。
- リツイート / 引用 RT 経由の表示、handle 変更には個別対応が要る。
- モバイル版 X は DOM 構造が PC 版と異なるため、セレクタの分岐が必要になる場合がある。
- 自動化は X の利用規約上グレー。**無料・個人用のクライアント側表示制御**に限定する。

## 7. 今後の検討

- 手動の非表示リスト編集 UI（収集の補完）。
- ホーム TL への適用範囲拡張（オプトイン）。
- マッチング判定ロジックのユニットテスト整備。
